#### 2.3 Unified route guard — `src/lib/authz/guard.ts`

```ts
export type GuardOptions = {
  capability?: Capability;
  entitlement?: EntitlementKey;
  recordUsage?: { kind: UsageKind; units?: number };
  allowAgentRail?: boolean;
};

export function withGuard<TArgs, TRet>(
  opts: GuardOptions,
  handler: (ctx: GuardContext, ...args: TArgs) => Promise<TRet>,
): (req: Request, ...args: TArgs) => Promise<Response>;
```

Guard execution order:

1. Resolve session via `requireSession()`
2. Assert capability (if provided)
3. Assert entitlement (if provided)
4. Execute handler
5. On success, record usage AFTER work succeeds — never bill for a failed call
6. Error mapping: `PlanUpgradeRequired` → 402 · `QuotaExceeded` → 429 · `AuthorizationDenied` → 403 · no session → 401

All guard hits write an `AccessAudit` row regardless of outcome.

### Milestone 6 — Agent rail authorization (Rail B)

**PR title:** `feat(agent-rail): agent identity, scopes, MCP, JWT auth`
**Goal:** same entitlement engine drives the agent endpoints. `AgentIdentity` maps to scopes → capabilities + entitlements.

#### 6.1 Agent JWT scheme

- Bearer JWT signed by the `AgentIdentity`'s private key (Ed25519 preferred). We hold only the public key.
- Required claims: `iss` (agent identity id), `aud` (`"margot-agent-rail"`), `exp` (≤ 10 min from `iat`), `sub` (companyId), `scope` (space-separated `AGENT_SCOPES` subset).
- Verification in `src/lib/agent-auth.ts` using Node's `crypto.subtle`. Do NOT introduce a JOSE library unless absolutely required (justify in PR if so).

#### 6.2 Identity issuance

- `POST /api/agent/v1/identity/register` — guard: `agent.identity.issue`. Plan must have `agentRailEnabled: true` (FREE/STARTER/STUDIO do not; PRACTICE, AGENT_DEV, AGENT_PRO, AGENT_SCALE, LLM_FEDERATION do).
- `DELETE /api/agent/v1/identity/:id` — guard: `agent.identity.revoke`.

#### 6.3 Agent endpoints

| Endpoint                       | Scope                                  | Entitlement                           | Usage recorded                             |
| ------------------------------ | -------------------------------------- | ------------------------------------- | ------------------------------------------ |
| `POST /api/agent/v1/query`     | `agent:read` + optional `agent:mode:*` | Mode entitlement on company plan      | `AGENT_READ` + `MODE_*_USED` if applicable |
| `POST /api/agent/v1/narrative` | `agent:narrative`                      | `narratives.generate` on company plan | `AGENT_NARRATIVE` + `NARRATIVE_GENERATED`  |
| `POST /api/agent/v1/synthesis` | `agent:synthesis`                      | —                                     | `AGENT_SYNTHESIS` + `CFO_TURN`             |
| `GET /api/agent/v1/usage`      | `agent:read`                           | —                                     | —                                          |

#### 6.4 Agent rail inside `withGuard`

When `allowAgentRail: true`: accept session cookie OR verified agent JWT. If agent JWT: resolve `companyId` from `sub`, load `AgentIdentity`, verify scope ⊇ required scopes, populate `GuardContext.agent`.

Scope → capability mapping in `src/lib/authz/agent-capability-map.ts`:

- `agent:read` / `agent:synthesis` → `cfo.chat`
- `agent:narrative` → `narratives.generate`
- `agent:mode:proposal` → `cfo.mode.proposal`

#### 6.5 Mode safety for agents

- Proposal mode requires `agent:mode:proposal` scope AND a verified `actsOnBehalfOf` claim in JWT (prevents impersonation of biz-dev rep).
- Board mode DENIED unless `AgentIdentity.issuedByUserId` was an ADMIN (track via new column on `AgentIdentity`).

#### 6.6 Rate limiting — `src/lib/agent-auth/rate-limit.ts`

In-memory token bucket. Default 60 req/min per `AgentIdentity`. Override via optional `rateLimitPerMinute` column (add to schema in this milestone). Add TODO to move to Redis for multi-process prod.
