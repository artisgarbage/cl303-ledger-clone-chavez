## Executive Summary

This audit reviewed the **Ledger** financial intelligence platform, which stores and processes real codelab303 LLC financial data alongside future multi-tenant company data. The platform integrates QuickBooks P&L data, generates AI narratives via Claude, and provides project profitability analytics.

**Key findings:**

- **No secrets leaked in git history** — verified via manual inspection and gitleaks scan
- **Strong baseline security posture** — `.gitignore` and `.dockerignore` properly exclude financial data and secrets; Dockerfile uses non-root user; CI runs gitleaks and blocks builds if `setup_data/` exists

**Executive summary:** 6 of 14 findings remediated (all P0 findings + 2 P1 findings). 8 open items remain (SEC-07 through SEC-14).

### Assets (ordered by sensitivity)

1. **codelab303 LLC financial data** — 2024-2026 P&L exports (`setup_data/*.xlsx`) containing real revenue, expenses, vendor/contractor names. **Current status:** No longer tracked in git, properly gitignored, excluded from Docker builds. ✅

3. **Anthropic API key** — grants access to Claude API; charges to codelab303's account; potential data egress channel. **Current status:** Env-var only, no hardcoded values. ✅
4. **NextAuth secret** — session signing key; compromise = session forgery. **Current status:** Env-var only, no hardcoded values. ✅
5. **Database** — PostgreSQL with all financial records, user credentials (bcrypt hashes), session tokens. **Protection:** TLS in transit (assumed), backups not yet configured.
6. **KMS keys** — (future) for encrypting tenant financials at rest. **Not yet implemented.**

2. **Next.js routes → Prisma queries** — `companyId` filtering is the tenant boundary (ad-hoc, not enforced)

4. **Next.js → PostgreSQL** — TLS assumed, credentials from env var

## Detailed Findings

### SEC-01: IDOR in `/api/admin/users/[id]` — cross-tenant user modification/deletion

**Severity:** P0  
**Category:** Multi-Tenancy  
**Effort:** Small (1 PR, ~30 lines)

**Evidence:**

```typescript
// src/app/api/admin/users/[id]/route.ts
export async function PATCH(
  req: NextRequest,
  { params }: { params: Promise<{ id: string }> },
) {
  const session = await requireAdmin();
  if (!session)
    return NextResponse.json({ error: "Forbidden" }, { status: 403 });

  const { id } = await params;
  // ⚠️ No companyId check before update!
  const user = await prisma.user.update({
    where: { id },
    data: updateData,
    // ...
  });
}
```

**Impact:**

- An admin from **Company A** can modify or delete users belonging to **Company B** by guessing/enumerating user IDs.
- Enables privilege escalation: attacker can promote their account to ADMIN in another tenant.
- Enables data destruction: attacker can delete all users from a competitor tenant.

**Recommendation:**

1. Add a preliminary check:
   ```typescript
   const target = await prisma.user.findUnique({
     where: { id },
     select: { companyId: true },
   });
   if (!target || target.companyId !== session.user.companyId) {
     return NextResponse.json({ error: "Not found" }, { status: 404 });
   }
   ```
2. Use `findFirst({ where: { id, companyId }})` or a compound `where` clause in the update/delete.
3. Add integration test: create two tenants, verify admin A cannot touch user B.

**Related:** This pattern should be audited across all `[id]` routes. Current audit shows `/api/people/[id]` and `/api/projects/[id]` correctly validate `companyId`; only the user routes are vulnerable.

### SEC-02: AI narratives send line-item names (vendor/customer PII) to Anthropic without opt-in or redaction

**Severity:** P0  
**Category:** AI Egress / Data Protection  
**Effort:** Medium (2 PRs, ~150 lines)

**Evidence:**

```typescript
// src/app/api/narratives/generate/route.ts
const topLineItems = allLineItems
  .sort((a, b) => Math.abs(b.amount) - Math.abs(a.amount))
  .slice(0, 20)
  .map((item) => ({
    category: item.category,
    name: item.name, // ⚠️ Could be "Acme Corp Invoice #1234" or "John Doe - Contractor"
    amount: item.amount,
  }));
```

**Impact:**

- QuickBooks line-item `name` field often contains customer names, vendor names, invoice numbers, or employee names.
- Sending this to Anthropic without explicit user consent violates GDPR/CCPA data processing principles.
- Anthropic's [Commercial Terms](https://www.anthropic.com/legal/commercial-terms) state they do **not** train on API data, but data is still transmitted and logged server-side.
- If a data subject (customer/vendor) requests "who has my data?", the answer now includes "Anthropic via Claude API."

**Recommendation:**

**Phase 1 (P0 — ship in this ticket):**

1. Add `CompanySettings.narrativesEnabled` boolean (already exists, default `true`) — change default to **`false`**.
2. Gate all narrative generation routes:
   ```typescript
   const settings = await prisma.companySettings.findUnique({
     where: { companyId },
   });
   if (!settings?.narrativesEnabled) {
     return NextResponse.json(
       { error: "Narratives disabled for this company" },
       { status: 403 },
     );
   }
   ```
3. Add UI toggle in Settings page: "Enable AI-generated financial narratives (sends aggregated financial data to Anthropic)."
4. Redact `topLineItems` before sending to Claude: replace `name` with generic labels or omit it entirely.
   ```typescript
   const topLineItems = allLineItems.slice(0, 20).map((item) => ({
     category: item.category,
     // name: "[REDACTED]",  // or omit entirely
     amount: item.amount,
   }));
   ```

**Phase 2 (P1 — follow-up issue):**

- Add granular redaction settings: "Include line-item names in AI narratives? (may expose vendor/customer names)"
- Log every narrative generation call to `IngestAudit` or a new `NarrativeAudit` table with: `companyId`, `userId`, `type`, `periodStart`, `periodEnd`, `dataHash` (SHA-256 of snapshot JSON).
- Document in `docs/security/ai-egress.md` exactly what data is sent to Anthropic and under what conditions.

**Related:** Verify Anthropic SDK error handling does not leak API key in stack traces (see SEC-02b below).

### SEC-03: `User.companyId` is nullable — creates orphaned users and auth bypass risk

**Severity:** P0  
**Category:** Multi-Tenancy / Auth  
**Effort:** Medium (1 migration, ~50 lines code change)

**Evidence:**

```prisma
// prisma/schema.prisma
model User {
  companyId String?  // ⚠️ Nullable
  company   Company? @relation(fields: [companyId], references: [id])
}
```

**Impact:**

- A user with `companyId = null` can:
  1. Authenticate successfully (NextAuth returns the session)
  2. Bypass tenant-scoping logic in routes that cast `companyId` as non-null
  3. Cause runtime errors in routes that assume `companyId` is always present
- Example vulnerable route:
  ```typescript
  const companyId = (session.user as { companyId: string }).companyId;
  // If companyId is actually null, this type assertion hides the bug
  ```

**Recommendation:**

1. **Make `companyId` non-null** in schema:
   ```prisma
   model User {
     companyId String
     company   Company @relation(fields: [companyId], references: [id])
   }
   ```
2. **Write migration** to backfill or delete orphaned users:

   ```sql
   -- Option A: Assign to a default "orphaned" company
   UPDATE "User" SET "companyId" = (SELECT id FROM "Company" WHERE name = 'Orphaned Users' LIMIT 1)
   WHERE "companyId" IS NULL;

   -- Option B: Delete orphans (safer if none should exist)
   DELETE FROM "User" WHERE "companyId" IS NULL;
   ```

3. **Update all auth code** to remove type assertions and rely on non-null type:
   ```typescript
   const companyId = session.user.companyId; // TypeScript knows this is string, not string | null
   ```
4. **Add constraint test:** Attempt to create a user without `companyId` and assert it fails.

**Risk:** If orphaned users exist in production, they must be handled before deploying the non-null migration. Review DB first.

### SEC-04: No audit logging for financial data access or mutations

**Severity:** P0  
**Category:** Audit / Compliance  
**Effort:** Medium (1 PR, ~120 lines)

**Evidence:**

- `IngestAudit` table exists for file uploads (good) but no audit trail for:
  - Read access to `/api/periods`, `/api/narratives/[id]`, `/api/people/[id]`
  - Mutations to `FinancialPeriod`, `Narrative`, `Person`, `Project`
- If a data breach occurs, there is no way to answer:
  - "Who accessed tenant X's financial data?"
  - "When was this record last modified, and by whom?"

**Impact:**

- **Compliance failure** for SOC 2, GDPR Article 30 (records of processing), HIPAA audit trails (if health data is ever added).
- **Forensics blindness:** Cannot trace insider access or malicious queries.
- **Customer trust:** Paying customers expect audit logs for financial data access.

**Recommendation:**

**Phase 1 (P0 — ship in this ticket):**

1. Add `AccessAudit` table:
   ```prisma
   model AccessAudit {
     id        String   @id @default(cuid())
     companyId String
     userId    String
     action    String   // "READ" | "CREATE" | "UPDATE" | "DELETE"
     resource  String   // "FinancialPeriod" | "Narrative" | "Person" | "Project"
     resourceId String?
     metadata  Json?    // { route, method, ip, userAgent }
     createdAt DateTime @default(now())
   }
   ```
2. Add middleware or helper function:
   ```typescript
   async function logAccess(
     companyId: string,
     userId: string,
     action: string,
     resource: string,
     resourceId?: string,
   ) {
     await prisma.accessAudit.create({
       data: { companyId, userId, action, resource, resourceId, metadata: {} },
     });
   }
   ```
3. Instrument **read-heavy routes** first:
   - `GET /api/narratives/[id]`
   - `GET /api/periods` (log with `action: "LIST"`)
   - `GET /api/people/[id]`
4. Do **not** log row contents or PII — only metadata (who, what, when).

**Phase 2 (P1 — follow-up issue):**

- Add admin UI: `/admin/audit-log` with filters (date range, user, resource type).
- Export to CSV for compliance reviews.
- Retention policy: archive logs > 1 year to cold storage.

### SEC-05: `xlsx` dependency has HIGH severity Prototype Pollution CVE

**Severity:** P1  
**Category:** Supply Chain  
**Effort:** Small (1 PR, dependency swap)

**Evidence:**

```
npm audit --omit=dev
xlsx
Severity: high
Prototype Pollution in sheetJS - https://github.com/advisories/GHSA-4r6h-8v6p-xvw6
No fix available
```

**Impact:**

- Prototype pollution allows attacker to inject properties into `Object.prototype`, potentially leading to:
  - Denial of Service (crash on certain property access)
  - Security bypass (e.g., `if (user.isAdmin)` can be poisoned)
  - RCE in some contexts (Node.js version-dependent)
- **Attack vector:** Malicious XLSX file uploaded via `/api/admin/ingest` or `/api/imports/quickbooks`.

**Recommendation:**

1. **Replace `xlsx` with `exceljs`** (actively maintained, no known CVEs):
   ```bash
   npm uninstall xlsx
   npm install exceljs
   ```
2. Update `src/lib/parsers/quickbooks.ts` (or equivalent) to use `exceljs` API.
3. Regression test: Parse a known QuickBooks P&L export and assert line items match.
4. **Interim mitigation** (if swap is too disruptive): Validate XLSX structure before parsing, reject files with suspicious property names (e.g., `__proto__`, `constructor`).

### SEC-06: `prisma db push` on every container start risks schema drift in production

**Severity:** P1  
**Category:** Container / Operations  
**Effort:** Small (1 PR, update entrypoint)

**Evidence:**

```sh
# docker-entrypoint.sh
node_modules/.bin/prisma db push --skip-generate
```

**Impact:**

- `prisma db push` auto-applies schema changes without migration files.
- In production, this can:
  - Silently drop columns if the Prisma schema is rolled back (e.g., deploy an old image).
  - Cause data loss if a dev accidentally deploys a WIP schema.
  - Break other services that depend on a stable schema (e.g., analytics ETL).
- **No rollback path** — migrations are not versioned or tracked.

**Recommendation:**

1. **Use `prisma migrate deploy`** in production:
   ```sh
   # docker-entrypoint.sh (prod)
   node_modules/.bin/prisma migrate deploy --skip-generate
   ```
2. Keep `prisma db push` for **local dev only** (document in README).
3. Add CI check:
   ```yaml
   - name: Verify migrations are up to date
     run: npx prisma migrate diff --from-schema-datamodel prisma/schema.prisma --to-schema-datasource $DATABASE_URL --exit-code
   ```

### SEC-07: NextAuth session cookie flags not explicitly hardened

**Severity:** P1  
**Category:** Auth  
**Effort:** Small (1 PR, ~10 lines)

**Evidence:**

```typescript
// src/lib/auth.config.ts
export const authConfig: NextAuthConfig = {
  session: { strategy: "jwt" },
  // ⚠️ No explicit cookie config
};
```

**Impact:**

- Default NextAuth cookie flags may not set:
  - `Secure` (HTTPS-only) — allows session theft over HTTP
  - `SameSite=Strict` — allows CSRF attacks
  - `HttpOnly` (default is true, but should verify)

**Recommendation:**

```typescript
export const authConfig: NextAuthConfig = {
  session: { strategy: "jwt" },
  cookies: {
    sessionToken: {
      name: `__Secure-next-auth.session-token`,
      options: {
        httpOnly: true,
        secure: true, // HTTPS only
        sameSite: "strict",
        path: "/",
      },
    },
  },
};
```

### SEC-10: CRON_SECRET has no fail-fast validation

**Severity:** P1  
**Category:** AuthZ  
**Effort:** Small (1 PR, ~5 lines)

**Evidence:**

```typescript
// src/app/api/narratives/scheduled/route.ts
const expectedSecret = process.env.CRON_SECRET;
if (!expectedSecret) {
  return NextResponse.json(
    { error: "CRON_SECRET not configured" },
    { status: 500 },
  );
}
```

**Impact:**

- If `CRON_SECRET` is not set, the endpoint **still accepts requests** (500 error, but continues).
- Should fail at **boot time**, not request time.

**Recommendation:**

```typescript
// src/lib/env.ts
if (!process.env.CRON_SECRET) {
  throw new Error("CRON_SECRET environment variable is required");
}
```
