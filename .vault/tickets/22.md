# TIK-013 — Milestone 1: Pricing & Entitlements (Backend-First)

**Issue:** https://github.com/artisgarbage/cl303-ledger-clone-chavez/issues/22
**Epic:** Full pricing, entitlements & authorization system
**Strategy:** `strategy/PRICING_AND_GTM.md` §4

## Goal

Build the billing metering substrate — plans, subscriptions, usage tracking, and entitlement enforcement. Pure backend work. No UI changes except error responses for gated features.

## Approach

Following the epic's Milestone 1 specification:

1. **Schema changes** — Add five new Prisma models:
   - `Plan` — product catalog (FREE, STARTER, STUDIO, PRACTICE, ENTERPRISE + 4 agent-rail plans)
   - `Subscription` — company → plan mapping with period tracking
   - `UsageEvent` — append-only log of metered actions (narratives, CFO turns, agent calls)
   - `OverageCharge` — settled overage billing records
   - `AgentIdentity` — agent-rail credentials (deferred to M2, stub only in M1)

2. **Entitlement library** — `src/lib/billing/entitlements.ts`:
   - `getEntitlements(companyId)` — resolve plan + capabilities
   - `assertEntitlement(companyId, key)` — throw if missing
   - `recordUsage(companyId, kind, units)` — write event + return running total
   - `computeOverage(companyId, periodStart, periodEnd)` — pure function

3. **Seed plans** — Populate all 9 plans (5 human-rail + 4 agent-rail) per pricing doc

4. **Wire into existing routes**:
   - `/api/narratives/generate` — check quota, record usage
   - `/api/cfo/chat` — check mode entitlement + quota, record usage
   - Both return 402 with `PlanUpgradeRequired` error shape on failure

5. **Tests** — Unit tests for entitlement logic, integration tests extending existing CFO flow tests

## Key Technical Decisions

1. **Entitlements stored as JSON** — Plan.entitlementsJson contains the full capability set. Simpler than normalized tables for v1, easy to version.

2. **FREE plan auto-assigned** — Companies without a Subscription row get FREE plan via fallback logic in `getEntitlements()`. Never write implicit subscription rows on read.

3. **Period boundaries** — Use subscription's `currentPeriodStart/End` if present; fall back to calendar month UTC for FREE plan (no subscription record).

4. **Overage vs. hard cap** — FREE plan hard-caps (402 response). STARTER+ allows overage; `OverageCharge` records written at period-close (M3).

5. **Mode entitlement check location** — `/api/cfo/chat` checks mode entitlement at request time based on plan, not at conversation-creation time. Allows seamless plan upgrades mid-conversation.

## Migration Strategy

1. Create migration with safeguards:
   - Add new tables (no FK conflicts — all nullable company refs initially)
   - Backfill companies: add `Subscription` row for each `Company` pointing to FREE plan
   - Make `Subscription.companyId` unique after backfill

2. Rollback plan:
   - Migration is additive — rollback drops 5 new tables
   - No data loss (existing tables untouched)

## Testing Strategy

- Plan resolution: company with subscription → correct plan
- Plan resolution: company without subscription → FREE
- Quota math: under cap, at cap, over cap
- Overage computation: various usage patterns
- Mode entitlement: each mode × each plan tier

**Integration tests** (extend `src/app/api/cfo/conversation-flow.test.ts`):

**No E2E needed for M1** — pure backend, no UI changes to test

## Error Response Shape

All 402 responses follow this contract:

```json
{
  "error": "PlanUpgradeRequired",
  "message": "You've reached the 5 narratives/month limit on the Free plan.",
  "code": "QUOTA_EXCEEDED",
  "requiredPlanSlug": "STARTER",
  "upgradeUrl": "/account/billing?upgrade=STARTER"
}
```

## Risks

1. **Seed timing** — Plan seed must run before entitlement checks. Mitigated: seed script is idempotent, runs on every `npm run db:seed`.

2. **Existing conversations breaking** — If plan enforcement is too aggressive, existing FREE users can't continue conversations. Mitigated: mode check is per-turn, not per-conversation; existing INTERNAL-mode conversations continue working.

3. **Migration on existing data** — Backfilling subscriptions for all companies at once. Mitigated: migration is in a transaction, small dataset (<100 companies in dev).

4. **Anthropic rate limiting** — Heavy testing could hit Anthropic rate limits. Mitigated: tests use mocked LLM responses where possible.
